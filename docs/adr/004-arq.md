# ADR 004: ARQ vs Celery for Background Tasks

## Status
Accepted

## Context
NEXUS requires background processing for tasks that are too slow to happen synchronously during an API request (e.g., resume parsing, LLM extraction, embedding generation). The original documentation suggested Celery.

## Decision
We will use **ARQ (Async Redis Queue)** instead of Celery for background job orchestration. 

## Consequences
- **Positive**: ARQ is native `asyncio`, which aligns perfectly with our async FastAPI, asyncpg, and async Qdrant/Redis stack.
- **Positive**: Less operational overhead. ARQ uses Redis directly without needing an extra message broker like RabbitMQ, and it doesn't require a separate Beat scheduler process.
- **Negative**: ARQ has a smaller ecosystem than Celery and lacks some advanced enterprise features (e.g., complex canvas workflows like chords/chains), but those are not required for our current linear background pipelines.
